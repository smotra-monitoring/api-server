# Agent Claiming Workflow

The server implements a **three-phase onboarding** workflow that lets agents register themselves and be securely provisioned with an API key by an administrator — without any shared secrets or pre-configuration on the agent.

---

## Overview

```
Agent                          Server                         Admin
  |                               |                              |
  |-- POST /v1/agent/register --> |                              |
  |<-- { status, pollUrl } -------|                              |
  |                               |                              |
  |                               |<-- POST /v1/agents/claim ----|
  |                               |    (claimToken + sectionId)  |
  |                               |                              |
  |-- GET  /v1/agent/{id}/        |                              |
  |        claim-status -------> |                              |
  |<-- { status: "claimed",       |                              |
  |      apiKey: "sk_live_..." }--|                              |
  |                               |                              |
  |   (key delivered once;        |                              |
  |    subsequent polls           |                              |
  |    return "pending") -------> |                              |
```

---

## Phase 1 — Agent Self-Registration

**Endpoint**: `POST /v1/agent/register`  
**Authentication**: None required.

The agent performs this step on first startup (or whenever it has no stored API key).

### What the agent must do before calling this endpoint

1. **Generate a UUIDv7 agent ID** — stable across restarts; store it locally.
2. **Generate a cryptographically random claim token** — at least 64 characters. Store the plaintext locally; never send it again after this step.
3. **Hash the token**: `SHA-256(claimToken)` — send only the hash to the server.

### Request

```bash
curl -X POST http://localhost:8080/v1/agent/register \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "019c1234-5678-7abc-def0-123456789abc",
    "claimTokenHash": "a3f5b2c8d1e9f4a7b6c5d2e1f8a9b4c7d3e8f1a2b5c9d4e7f0a3b6c1d2e5f8a9",
    "hostname": "prod-web-01",
    "agentVersion": "1.0.0"
  }'
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `agentId` | UUIDv7 | Yes | Stable agent identity, generated by the agent |
| `claimTokenHash` | hex string | Yes | SHA-256 hash of the plaintext claim token |
| `hostname` | string | Yes | Hostname of the machine running the agent |
| `agentVersion` | string | Yes | Agent software version |

### Response — 201 Created

```json
{
  "status": "pending_claim",
  "pollUrl": "/v1/agent/019c1234-5678-7abc-def0-123456789abc/claim-status",
  "claimUrl": "https://yoursite.com/claim"
}
```

The agent should begin polling `pollUrl` (see Phase 3). The `claimUrl` is shown to the user in the web UI where an admin can claim the agent.

**Re-registration**: if the agent calls register with the same ID again before being claimed, the existing pending claim is replaced.

---

## Phase 2 — Administrator Claiming

**Endpoint**: `POST /v1/agents/claim`  
**Authentication**: OAuth2 bearer token (admin; user context extraction pending — currently unenforced).

An administrator reviews the list of pending agents in the web UI and claims one by providing the original plaintext claim token (which the agent displayed on its console or stored in its local config).

### Request

```bash
curl -X POST http://localhost:8080/v1/agents/claim \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <admin-token>" \
  -d '{
    "agentId": "019c1234-5678-7abc-def0-123456789abc",
    "claimToken": "original-plaintext-token-from-agent",
    "sectionId": "019c9876-5432-7fed-cba0-987654321fed",
    "name": "Production Web Server 01"
  }'
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `agentId` | UUIDv7 | Yes | Agent to claim |
| `claimToken` | string | Yes | Plaintext claim token (not the hash) |
| `sectionId` | UUIDv7 | Yes | Section (group) the agent belongs to |
| `name` | string | No | Human-readable display name for the agent |

### What the server does

1. Looks up the pending claim for `agentId`.
2. Computes `SHA-256(claimToken)` and compares it in constant time against the stored hash.
3. Checks the claim has not expired (`claim_token_expiration_hours` in config).
4. Creates the agent record in the `agents` table.
5. Generates a cryptographically random API key (32+ bytes).
6. Stores `SHA-256(apiKey)` in `agents.api_key_hash`.
7. Stores the **plaintext** key in `agent_claims.api_key_plaintext` — one-time delivery storage.
8. Marks the claim as `claimed`.

### Response — 200 OK

```json
{
  "status": "claimed"
}
```

## Phase 3 — API Key Delivery

**Endpoint**: `GET /v1/agent/{agentId}/claim-status`  
**Authentication**: None required.

The agent polls this endpoint after registration. Polling uses linear backoff configured in `agent.claim_poll_*` config fields.

### Response — Pending (waiting for admin)

```json
{
  "status": "pending"
}
```

### Response — Claimed (first poll after claiming)

```json
{
  "status": "claimed",
  "apiKey": "..."
}
```

**One-time delivery**: the server clears `api_key_plaintext` from the database immediately after returning it. Any subsequent poll for the same agent returns `{ "status": "pending" }` — the key is gone from the server.

The agent must save the API key to its local persistent storage immediately upon receipt.

### Response — Expired claim

```json
{
  "status": "expired"
}
```

The claim token has passed `claim_token_expiration_hours`. The agent must re-register.

---

## Security Properties

| Property | Mechanism |
|----------|-----------|
| Claim token never sent in plaintext to server | Agent sends `SHA-256(token)`; admin sends plaintext only at claim time |
| Timing-safe comparison | `crypto/subtle.ConstantTimeCompare` used for all hash comparisons |
| API key never persisted long-term in plaintext | Cleared from DB immediately after first delivery |
| Claim expiry | Configurable; server rejects claims older than `claim_token_expiration_hours` |
| No shared secrets required | Agent bootstraps with a self-generated token; no pre-provisioning needed |

---

## Configuration

Claim-related configuration lives under the `agent` key:

```yaml
agent:
  server_url: https://api.example.com
  claim_token_expiration_hours: 24

  # Linear backoff for claim-status polling
  claim_poll_initial_interval_secs: 5
  claim_poll_increment_secs: 5
  claim_poll_max_interval_secs: 60
```

| Field | Description |
|-------|-------------|
| `claim_token_expiration_hours` | How long a pending claim is valid (min 1) |
| `claim_poll_initial_interval_secs` | First polling interval |
| `claim_poll_increment_secs` | Added to interval after each poll |
| `claim_poll_max_interval_secs` | Maximum polling interval (must be ≥ initial) |

## Future Work

- **OAuth2 admin context**: the claim endpoint currently accepts any bearer token. Once OAuth2 user context extraction is implemented, the server will validate the admin's tenant membership and verify that `sectionId` belongs to their tenant.
- **Metrics**: Prometheus counters for `agent_register`, `agent_claim`, and `agent_claim_status` handlers are tracked in the ROADMAP.
- **Rate limiting**: registration and claim-status polling endpoints are candidates for per-IP rate limiting to prevent enumeration.
